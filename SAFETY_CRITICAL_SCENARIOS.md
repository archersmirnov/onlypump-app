# Критичные сценарии проверки OnlyPump

Дата: 2026-07-01

Цель: перед важными правками и после них быстро проверять сценарии, где регрессии обычно всплывают не сразу: сохранение, повтор, редактирование, аналитика, виджеты и расчеты по датам.

Этот файл не меняет поведение приложения. Это ручной safety-checklist.

## Как пользоваться

1. Перед правкой выбрать домен, который может быть затронут.
2. Выполнить короткий pre-check из этого файла.
3. Сделать минимальную правку.
4. Повторить тот же сценарий после правки.
5. Если что-то упало, смотреть:
   - `window.__ONLYPUMP_DATA_INVARIANTS__`
   - `window.__ONLYPUMP_DOMAIN_EVENTS__`
   - консольные предупреждения `[OnlyPumpDataEntry]`
   - консольные события `[OnlyPumpDomainEvent]`, если включен `window.__ONLYPUMP_DEBUG_EVENTS__ = true`

## Уровни проверки

### Быстрая проверка

Использовать после мелких UI-правок.

- приложение открылось без ошибок в консоли;
- текущий экран не потерял данные;
- переключение между 2-3 основными экранами работает;
- нет новых предупреждений `[OnlyPumpDataEntry]`;
- `window.__ONLYPUMP_DATA_INVARIANTS__` пустой или содержит только уже известные предупреждения.

### Средняя проверка

Использовать после правок в общих данных, расчетах, карточках, графиках, календарях.

- пройти быстрый сценарий;
- проверить один сценарий записи данных;
- перезагрузить приложение;
- убедиться, что данные остались;
- проверить один связанный виджет или график.

### Полная проверка

Использовать после правок в тренировках, питании, профиле, API, нормализации данных.

- пройти все сценарии для затронутого домена;
- проверить связанные виджеты;
- проверить аналитику;
- проверить перезагрузку;
- проверить, что база и UI показывают одно и то же.

## Тренировки

### Создание

- создать новую тренировку;
- добавить упражнение;
- добавить подход;
- изменить вес;
- изменить повторы;
- изменить RIR/RPE, если доступны;
- перезагрузить приложение;
- проверить, что тренировка, упражнение и подход остались.

Ожидаемые события:

- `workouts.persistence.request`
- `workouts.persistence.success`

Красный флаг:

- после перезагрузки тренировка пропала;
- подходы вернулись после удаления;
- `workouts.persistence.failed`;
- в базе есть workout, но нет exercises/sets.

### Повтор

- открыть прошлую тренировку;
- нажать повтор;
- выбрать вариант "сейчас";
- изменить хотя бы один подход;
- завершить тренировку;
- перезагрузить приложение;
- проверить, что новая тренировка сохранена отдельно от старой.

Отдельно проверить:

- повтор как уже выполненную тренировку;
- повтор запланированной тренировки;
- повтор выполненной тренировки.

Красный флаг:

- повтор создался, но последующие изменения не сохраняются;
- новая тренировка использует старые id подходов;
- удаленный подход возвращается.

### Редактирование сохраненной тренировки

- открыть completed-тренировку;
- поменять название;
- поменять вес/повторы;
- удалить подход;
- удалить упражнение;
- добавить новый подход;
- перезагрузить приложение;
- проверить, что все изменения остались.

Красный флаг:

- визуально изменение было, но после reload вернулось старое значение;
- completed-тренировку нельзя редактировать;
- active/completed статус расходится между UI и базой.

### Удаление

- удалить active-тренировку;
- удалить planned-тренировку;
- удалить completed-тренировку, если UI это разрешает;
- перезагрузить приложение;
- проверить, что удаленные записи не вернулись.

Красный флаг:

- тренировка исчезает визуально, но возвращается после reload;
- в UI удаление прошло, но в базе запись осталась активной;
- удаление затронуло тренировки другого пользователя.

### Виджеты и аналитика тренировок

- проверить объем недели;
- проверить плановый объем;
- проверить фактический объем;
- проверить тоннаж;
- проверить графики/аналитику тренировок.

Правило:

- плановые виджеты могут учитывать `planned`;
- фактическая аналитика должна учитывать только реально выполненные тренировки;
- если экран показывает "ожидаемый итог недели", он может учитывать planned + completed, но это должно быть явно.

## Питание

### Классический режим

- добавить продукт по граммам;
- изменить количество;
- удалить продукт;
- переключить дату;
- вернуться на дату добавления;
- перезагрузить приложение.

Ожидаемые события:

- `nutrition.action.started`
- `nutrition.api.request`, если доступен remote;
- `nutrition.action.success`

Красный флаг:

- еда пропала после reload;
- дневные итоги не совпадают с продуктами;
- запись появилась не в той дате.

### Правило ладони

- добавить белковый продукт через ладонь;
- добавить углевод через горсть;
- добавить овощ через кулак;
- добавить жир через большой палец;
- убедиться, что в режиме ладони не показываются граммы/ккал при добавлении;
- перезагрузить приложение;
- проверить дневные итоги.

Красный флаг:

- неизвестному продукту доступна ладонь;
- рис получает ладонь;
- масло получает ладонь;
- внутренний пересчет не попал в дневные итоги.

### TDEE/TEF

- добавить питание за сегодня;
- добавить питание за прошлую дату;
- открыть TDEE;
- проверить TEF для каждого дня;
- переключить столбики/календарь;
- проверить, что порядок дней не меняется от выбора даты.

Красный флаг:

- прошлые дни берут TEF из текущего дня;
- выбранный день становится последним без причины;
- свайп пересобирает окно вокруг selectedDate.

## Профиль и замеры

### Профиль

- изменить цель;
- изменить параметры тела;
- сохранить;
- перезагрузить приложение;
- проверить профиль, питание, TDEE и виджеты.

Ожидаемые события:

- `profile.save.started`
- `profile.api.request`, если доступен remote;
- `profile.save.success`

Красный флаг:

- профиль визуально сохранился, но после reload откатился;
- изменение профиля сломало цели питания или TDEE.

### Замеры

- сохранить вес;
- сохранить несколько замеров;
- переключить дату;
- открыть аналитику;
- открыть графики;
- перезагрузить приложение.

Ожидаемые события:

- `measurement.weight.save.started`
- `measurement.weight.save.success`

Красный флаг:

- карточка показывает одно, график другое;
- таблица видит замер, а график пустой;
- текущий объект `measurements` расходится с историей `measurementRecords`.

## Здоровье

- сохранить шаги;
- сохранить сон;
- сохранить воду;
- переключить дату;
- проверить главную;
- проверить аналитику;
- перезагрузить приложение.

Ожидаемые события:

- `health.log.save.started`
- `health.log.save.success`

Красный флаг:

- данные сохранились не в ту дату;
- главная видит данные, аналитика нет;
- TDEE берет шаги/сон из другого дня.

## Аналитика ученика для тренера

- открыть ученика;
- открыть "Обычный вид";
- открыть "Таблица";
- открыть "Тренировки";
- открыть "Графики";
- проверить скролл;
- открыть отдельный график;
- переключить период неделя/месяц/6 месяцев/год.

Красный флаг:

- карточка графика показывает данные, а модалка нет;
- месяц показывает данные, год показывает только один месяц;
- длинный период сжимает графики до нечитаемого состояния;
- вкладка "Графики" не скроллится.

## Минимальная проверка после любой правки

- открыть приложение;
- проверить консоль;
- выполнить одно действие записи в затронутом домене;
- перезагрузить приложение;
- проверить, что запись осталась;
- посмотреть `window.__ONLYPUMP_DOMAIN_EVENTS__`;
- посмотреть `window.__ONLYPUMP_DATA_INVARIANTS__`.

## Когда нельзя продолжать правку

Остановиться и сначала диагностировать, если:

- появилась ошибка сохранения;
- данные исчезают после reload;
- данные одного пользователя видны/удаляются у другого;
- active/completed/planned статусы расходятся;
- `window.__ONLYPUMP_DATA_INVARIANTS__` показывает новую проблему;
- в журнале есть `*.failed` по затронутому домену.
